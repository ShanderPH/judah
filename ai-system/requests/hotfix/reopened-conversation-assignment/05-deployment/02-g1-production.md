## Migration recovery

The first API attempt failed safely before promotion because the runtime role
could not create objects in `public`. A temporary schema `CREATE` grant was
tested, but the second attempt also failed safely because the runtime is not the
owner of existing support tables. The grant was revoked.

Migrations `0020` through `0023` were then generated by Django and applied by
the privileged Supabase migration executor. Their Django migration records were
written in the same migration operations. The application runtime retains
`CREATE=false`; the cycle table exists and all four migrations are recorded.

No backfill, HubSpot owner mutation, incident repair, canary or enforcement was
executed.
